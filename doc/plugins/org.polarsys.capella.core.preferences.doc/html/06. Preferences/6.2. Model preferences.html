MODEL PREFERENCES

Preferences are available from the Window / Preferences menu. A section is dedicated to Capella.


GENERAL

The “Check model version at load time” provides a security when opening a model. It checks whether the model has been produced with the current version and thus whether it needs to be migrated or not. The drawback of activating this check is that the opening of models takes twice as long.

The AIRD Fragment File Extension option allows specifying the expected file suffix for AIRD fragments.

It is strongly recommended to use “airdfragment”.

DELETE

On the sub section of Capella / Delete preferences, it is possible to specify deletion options:

  • If “Confirm deletions” is activated, then a confirmation dialog will be prompted for each deletion attempt. This dialog show a live impact analysis of the deletion.
  • If “If reusing components is allowed, synchronize …” is activated, then when deleting a Component, all the Parts that are typed by this component will be deleted as well.

Protected Elements: It is possible to prohibit the deletion of certain critical elements. In that case, the deletion command will not be active for these elements.

The following figure illustrates the prompted dialog when deleting the System Actor “Mission Control Operator”. All the elements in red will be deleted if the deletion is confirmed. The blue elements are the references towards the elements that are going to be deleted.

MODEL

On the sub section Model of Capella preferences, it is possible to specify global behavior regarding the management of specific elements.

When “Multiple inheritance allowed” is activated, a Class can inherit from several other Classes. Otherwise, it can only have one inheritance ancestor.

When “Multiple deployment allowed” is activated, one given Behaviour Physical Component can be deployed on several Node Physical Components.

When “Delete related associations when a class becomes primitive” is activated, existing associations towards the class which primitive status has changed is deleted automatically. If this option is not activated, then model validation is such cases will raise an error and the end user will be able to manually delete the obsolete association.

The three synchronization options define whether Port allocations and delegations are automatically set when performing allocations from editors. Activating this option helps improving the productivity.


OVERVIEW

Both options allow choosing whether activities related to Functional Analysis and Interface Engineering should be displayed or not (See MDE in Thales / Modeling Approaches / Modeling Strategies).

The sub sections of the Overview Preferences allow specifying individually which sections have to be displayed for each engineering phase.

PROJECT EXPLORER

When the “Project” option is activated, the semantic element “Project” is displayed (see illustration below). By default, this option is not active. Displaying the “Project” model element is useful

  • If there is a need to manage several system models in the same Capellamodeller resource (not really common),
  • In order to modify configuration property values located on the Project element (reusable components or not, default values for progress monitoring, etc.)

By default, this option should be deactivated.

REFINEMENT

This section is dedicated to delegation-based refinement of interface sequence diagrams. It is not relevant when exploiting Functional Analysis.

The checkbox “Allow diagram creation” specifies whether a diagram shall be automatically created for the target scenario of the refinement.

The checkbox “Allow diagram opening” specifies whether the diagram corresponding to the target scenario of the refinement shall be automatically opened or not.

The checkbox “Perform a pre-validation” specifies whether the refinement tool shall perform a set of elementary check verifying the source scenario are ready for refinement. It is strongly recommended to activate this option as well as the following one “Stop Refinement on pre-validation error”.

SCM

This section is related to the behaviour of Capella regarding the Software Configuration Management tool (e.g. Clearcase). The configuration option allows setting an automatic synchronization delay with the File System.

Activating this configuration option is mandatory when working with Clearcase Dynamic Views, in order to be notified immediately of changes performed by other users.

TRANSITIONS/GENERATION

It is possible to specify the behaviour of the automated transitions.

  • The checkbox “Use STANDARD Component Port rather than FLOW Component Port” specifies whether generation of component exchanges and interfaces generate STANDARD Component Port rather than FLOW Component Port.
  • The checkbox “Generate Component Port while generating Interfaces” specifies whether generation of interfaces from allocated functions generates providing/requiring Component Port rather than implementation/usage links.
  • The radio buttons “Propagate logical component breakdown in physical architecture” and “Propagate only leaf logical components in physical architecture” specifies whether the breakdown of Logical Components is kept at physical level or not.
  • The checkbox “Transit interfaces handled by Components from context/logical layers to physical architecture” specifies whether interfaces are duplicated from Logical architecture to Physical architecture when performing a Logical Component to Physical Component transition. If not activated, Physical Components will directly implement / use / provide / require interface located in the system analysis and logical architecture.
  • The checkbox “Transit allocated Exchange Items while transition of Interfaces, Functions or Components” specifies whether Interfaces, Functions or Components transition triggers transition of related Exchange Items (i.e. duplicates Exchange Items from one engineering phase to the following one). If not activated, Interface, Functions and Component Exchanges in the target engineering phase will still refer the Exchange Items located in the previous engineering phase.
  • The checkbox “Transit Data Types handled by Exchange Items while transition of Exchange Items” specifies whether Exchange Items transition triggers transition of related Data (i.e. duplicates Data from one engineering phase to the following one). If not activated, Exchange Items in the target engineering phase will still refer the Data elements located in the previous engineering phase.
  • The checkbox “Transit State Machines owned by a Component while transition of Components” specifies whether State Machines transition is triggered when performing a Component transition.
  • The checkbox “Transit Functional elements allocated by an Operational Entity or Operational Actor while their transition” specifies whether Operational Activities are transformed into System Functions when performing a transition of Operational Entities or Actors.
  • The checkbox “Include involved elements” on the Property Values section specifies whether involved elements will be propagated while a transition of Property Values
