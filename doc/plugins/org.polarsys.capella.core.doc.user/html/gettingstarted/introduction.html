Capella Introduction

A central activity of system engineering is the definition of both the architectures of the system and its software.
Capella is a tool intended to improve the quality of these architectures.

System / Software Architecture Definition

Defining a system architecture means

  • Defining and refining use cases
  • Performing and refining a functional analysis
  • Defining components realizing the functions of the system
  • Defining the interfaces between these components
  • Model the dynamics of the system with scenarios
  • Take into account non functional constraints (Performance, Safety, Human Factors, etc.)

A Methodological Guide

In Thales, methodologies exist to help and guide the various actors of the engineering process. It is notably the case of

  • Sys-EM, which defines the successive stages of the engineering process
  • Arcadia, which compliments Sys-EM with a brick specifically dedicated to the functional analysis and architecture definition.

Capella leverages these methodologies and embeds them in a tool which is helping and guiding the system engineer in its activity. Is also provides Thales-wide language and tools, very close to (and interoperable with) industry standards such as UML and Sys-ML. This ensures an better understanding between Thales engineers from all domains.

Principles

As an input, the system is seen and defined as a "black box", on the basis of system requirements and operational scenarios. Capella is intervening at the "white box" level by allowing to specify how the internal components of the system are going to fulfill the requirements. This process includes several activities:

  • Refinement of the functional analysis (enrichment of the functional trees and dataflows)
  • Re-use or creation of components,
  • allocation of the functions to components,
  • definition of the interfaces.

Capella supplies tools facilitating the daily work of the system engineer (transitions between architectures, filters and layers on diagrams, modeling acceleration tools, synthetic business view of relationships between model elements, etc.) and guarantying the quality of the designed architectures (impact analysis, model validation, etc.)
